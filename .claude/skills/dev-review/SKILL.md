# Dev Review

各フェーズで呼び出されるレビュースキル。
フェーズを指定して呼び出すことで、観点を絞り**コンテキスト消費を最小化**する。

## 呼び出し方

```
/dev-review <フェーズ名>
```

フェーズ名: `requirements` / `design` / `implement` / `test` / `pre-deploy`

---

## フェーズ別レビュー観点

### requirements（要件レビュー）
確認対象：要件定義書

| 観点 | チェック内容 |
|------|------------|
| 完全性 | Must要件に漏れはないか |
| 実現可能性 | 技術・期間・予算で達成できるか |
| 矛盾 | 要件間で矛盾・競合がないか |
| スコープ | 対象外が明確に定義されているか |

出力上限：**箇条書き10行以内**

---

### design（設計レビュー）
確認対象：設計書（アーキテクチャ・DB・API）

| 観点 | チェック内容 |
|------|------------|
| 要件整合 | 要件定義書の内容をすべてカバーしているか |
| スケーラビリティ | 利用規模に対してオーバー/アンダーエンジニアリングでないか |
| セキュリティ | 認証・認可・データ保護の設計があるか |
| 変更困難箇所 | 後から変えにくい決定に見落としはないか |

出力上限：**箇条書き10行以内**

---

### implement（実装レビュー）
確認対象：git diff / PR

```bash
git diff main...HEAD --stat
git diff main...HEAD
```

| 観点 | チェック内容 |
|------|------------|
| 正確性 | バグ・ロジック誤り・エッジケース漏れ |
| セキュリティ | OWASP Top10・入力検証・秘密情報の扱い |
| 設計整合 | 設計書との乖離がないか |
| 可読性 | 命名・複雑度・コメントの適切さ |

出力上限：**指摘事項のみ、1件3行以内**

---

### test（テストレビュー）
確認対象：テスト結果レポート・テストコード

| 観点 | チェック内容 |
|------|------------|
| カバレッジ | 重要ロジックが80%以上カバーされているか |
| 異常系 | エラー・境界値・権限のテストがあるか |
| 信頼性 | フレイキーなテストが混入していないか |

出力上限：**箇条書き8行以内**

---

### pre-deploy（デプロイ前レビュー）
確認対象：デプロイ計画・チェックリスト

| 観点 | チェック内容 |
|------|------------|
| チェックリスト | 全項目がチェック済みか |
| ロールバック | 手順が明確で即実行可能か |
| 影響範囲 | 既存機能への副作用がないか |
| タイミング | 低トラフィック時間帯か |

出力上限：**GO/NO-GO + 理由3行以内**

---

## 出力フォーマット（全フェーズ共通）

```
## レビュー結果: [フェーズ名]

判定: APPROVE ✅ / REQUEST CHANGES ⚠️ / REJECT ❌

### 必須修正（REQUEST CHANGES / REJECT のみ）
- （問題点と修正方針を1行で）

### 確認推奨（任意）
- （軽微な提案を1行で）

### 良かった点
- （1〜2点）
```

## コンテキスト節約ルール
- 差分は `--stat` で全量把握してから必要なファイルだけ読む
- 問題がない観点は「〇〇: 問題なし」の1行で済ませる
- 長い説明より「ファイル:行番号 — 要点」形式を使う
- 既に前フェーズで確認済みの内容は再確認しない
