# Dev Design

要件定義書をもとにシステム設計を行うスキル。

## 入力

`args` または前フェーズから引き継いだ要件定義書。

## ステップ 1: アーキテクチャ設計

技術スタックとシステム構成を決定する。

- **構成パターン選定**（モノリス／マイクロサービス／サーバーレスなど）
- **技術スタック確定**（フレームワーク、DB、インフラ）
- **外部連携**（API、認証基盤、決済など）

選定理由を必ず添える。複数案がある場合は比較表を出す。

## ステップ 2: データ設計

- エンティティ抽出とER図（Mermaid記法）
- テーブル定義（カラム名・型・制約・インデックス）
- データフロー

## ステップ 3: API設計

RESTまたはGraphQLのエンドポイント一覧：
- メソッド・パス・説明
- リクエスト／レスポンス例
- 認証・エラーコード

## ステップ 4: 画面設計（フロントエンドがある場合）

- 画面一覧と遷移図（Mermaid記法）
- 各画面のコンポーネント構成
- 状態管理方針

## ステップ 5: 設計書の出力

```markdown
# 設計書

## 1. システム構成図
## 2. 技術スタック
## 3. データ設計（ER図・テーブル定義）
## 4. API設計
## 5. 画面設計・遷移図
## 6. セキュリティ設計
## 7. インフラ構成
## 8. 懸念事項・技術的負債リスク
```

ユーザーの承認を得てから実装フェーズへ。

## スタイルガイド
- 設計の選択肢が複数ある場合は推奨案を1つ提示し、理由を添える
- 「あとで変えられない決定」は特に明示してユーザーに確認を取る
- Mermaid 記法でダイアグラムを積極的に使う

## インフラ設計の注意点（Docker Compose）

docker-compose.yml を設計・記述する際は以下を守る。

### Dockerイメージはバージョン固定する
`latest` タグは Docker Desktop のレイヤーDB破損等で pull 失敗するリスクがある。
必ずバージョンを明示する。

```yaml
# NG
image: dpage/pgadmin4:latest

# OK
image: dpage/pgadmin4:8.14
```

このルールはすべてのサービス（pgAdmin、Redis、Nginx等）に適用する。
バージョンは設計時点で動作確認済みのものを記載し、理由をコメントで残すと望ましい。
