# Dev Deploy

承認済みコードを環境にデプロイするスキル。

## 入力

`args` または前フェーズから引き継いだPR番号・デプロイ先情報。

## ステップ 1: デプロイ前チェック

以下を必ず確認してからデプロイを開始する：

- [ ] テストがすべてパスしている
- [ ] レビューが承認されている
- [ ] マイグレーションの有無と内容確認
- [ ] 環境変数・シークレットの設定確認
- [ ] ロールバック手順の確認
- [ ] デプロイ時間帯（本番は低トラフィック時間帯を推奨）

問題があれば即座にユーザーに報告してデプロイを停止する。

## ステップ 2: ステージング環境デプロイ（推奨）

本番デプロイの前にステージングで動作確認する：

```bash
# ステージングへのデプロイ（環境に応じて変更）
git merge <branch>
# CI/CDパイプラインのトリガーまたは手動デプロイコマンド
```

ステージングでの確認項目：
- 主要機能の動作確認（スモークテスト）
- ログにエラーが出ていないか
- マイグレーションが正常に完了したか

## ステップ 3: 本番デプロイ

ステージング確認OKの後、ユーザーに本番デプロイの許可を取る。

```bash
# 本番ブランチへのマージ
git checkout main
git merge --no-ff <branch>
git tag v<version>
git push origin main --tags
```

## ステップ 4: デプロイ後確認

デプロイ完了後、以下を確認する：
- ヘルスチェックエンドポイントの応答
- エラーレートの監視（5分間）
- 主要機能のスモークテスト

問題が発生した場合は即座にロールバックを実行する：
```bash
git revert HEAD
# または前バージョンのタグに戻す
```

## ステップ 5: 完了報告

```markdown
# デプロイ完了報告

## デプロイ情報
- 環境: staging / production
- バージョン: vX.X.X
- デプロイ日時: YYYY-MM-DD HH:MM
- デプロイ内容: （変更概要）

## 確認結果
- ヘルスチェック: OK / NG
- スモークテスト: PASS / FAIL
- エラーレート: 正常範囲内 / 異常あり

## URL
- 本番: https://...
- ステージング: https://...

## ロールバック手順
（万一の際の手順）
```

## スタイルガイド
- 本番デプロイは必ずユーザーの明示的な許可を得てから実行する
- 「たぶん大丈夫」でデプロイしない。不安な点は必ず確認する
- デプロイ後5分は監視を怠らない
