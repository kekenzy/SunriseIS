# Dev Test

実装コードを検証するスキル。

## 入力

`args` または前フェーズから引き継いだPR・ブランチ情報。

## ステップ 1: テスト計画

テスト対象と範囲を確認し、テスト計画を立てる：

| テスト種別 | 対象 | ツール例 |
|-----------|------|---------|
| ユニットテスト | 関数・クラス単体 | Jest, pytest, RSpec |
| 統合テスト | モジュール間連携 | Supertest, TestContainers |
| E2Eテスト | ユーザーシナリオ | Playwright, Cypress |
| 負荷テスト | パフォーマンス要件 | k6, Locust |

## ステップ 2: テストケース設計

各機能について以下の観点でテストケースを設計する：
- **正常系**：期待通りの入力・操作
- **異常系**：不正入力・エラー条件
- **境界値**：最大・最小・空・null
- **権限**：認証・認可の確認

## ステップ 3: テスト実装・実行

```bash
# 依存関係インストール
# テスト実行コマンド（プロジェクトに応じて）
npm test / pytest / bundle exec rspec
```

カバレッジ目標：重要ビジネスロジックは80%以上。

## ステップ 4: 結果レポート

```markdown
# テスト結果レポート

## サマリー
- 合計テスト数: X件
- 成功: X件 / 失敗: X件 / スキップ: X件
- カバレッジ: X%

## 失敗したテスト
（失敗したテストケースと原因）

## リスク・懸念事項
（テストできなかった範囲・手動確認が必要な箇所）

## 合否判定
PASS / FAIL
```

FAILの場合は実装フェーズに差し戻す。

## スタイルガイド
- テストは実装コードより先に書く（TDD）を推奨するが強制しない
- 「テストが通ればOK」ではなく「テストが正しいか」も確認する
- フレイキーテストは原因を特定してから修正する
