# Dev Manager

開発プロジェクト全体を管理するスキル。
各フェーズ後に `/dev-review` を自動実行し、品質を担保しながら完遂する。

## 開始時の必須アクション

**最初に以下を実行してグローバルポリシーを読み込む：**

```bash
cat ~/.claude/data/dev-global-policy.md
```

読み込んだポリシーを全フェーズで厳守する。プロジェクト固有の指示がポリシーと競合する場合はユーザーに確認する。

## AIエンジン使い分け方針（コンテキスト最適化）

| フェーズ | 推奨モデル | 理由 |
|---------|-----------|------|
| 要件定義 | Haiku | 選択式Q&A・定型フォーマット |
| 要件レビュー | Haiku | チェックリスト確認のみ |
| 設計 | Sonnet | アーキテクチャ判断・複数案比較 |
| 設計レビュー | Haiku | 観点が明確で短い出力 |
| 実装 | Sonnet | コード生成・ロジック実装 |
| 実装レビュー | Sonnet | diff読解・セキュリティ判断 |
| テスト | Sonnet | テスト設計・コード生成 |
| テストレビュー | Haiku | レポート確認・数値チェック |
| デプロイ前レビュー | Haiku | チェックリスト確認 |
| デプロイ | Sonnet | 手順実行・障害判断 |

> モデル切り替えは `/model haiku` または `/model sonnet` で行う。
> 判断に迷ったら Sonnet を使う。Opus は複雑な設計判断のみ。

## 進行フロー

```
[要件] → [要件レビュー] → [設計] → [設計レビュー]
→ [実装] → [実装レビュー] → [テスト] → [テストレビュー]
→ [デプロイ前レビュー] → [デプロイ]
         ↑___________________________|（差し戻し時）
```

## ステップ 1: プロジェクト把握

`args` からプロジェクト概要を受け取る。確認事項：
- プロジェクト名・目的
- 既存コードベースの有無
- 技術スタック指定の有無

## ステップ 2: フェーズ実行（レビュー込み）

各フェーズは以下の手順で進める：

```
1. フェーズ開始を宣言: 「▶ [フェーズ名] を開始します」
2. 該当スキルを呼び出す
3. 成果物が出たら /dev-review <フェーズ名> を呼び出す
4. APPROVE → 次フェーズへ
   REQUEST CHANGES → 同フェーズを修正して再レビュー
   REJECT → 前フェーズに差し戻し（理由をユーザーに報告）
```

### フェーズ一覧

| フェーズ | スキル | レビュー | 主な成果物 |
|---------|--------|---------|----------|
| 要件定義 | /dev-requirements | /dev-review requirements | 要件定義書 |
| 設計 | /dev-design | /dev-review design | 設計書 |
| 実装 | /dev-implement | /dev-review implement | PR |
| テスト | /dev-test | /dev-review test | テスト結果 |
| デプロイ | /dev-deploy | /dev-review pre-deploy | デプロイ完了 |

## ステップ 3: コンテキスト節約ルール

- 各フェーズの成果物は**要約（箇条書き5行以内）**を次フェーズに引き継ぐ（全文ではない）
- レビュー結果は判定と必須修正のみを記録する
- 同じ情報を繰り返し参照しない（一度読んだらメモして再読しない）
- フェーズが進むにつれて古いフェーズの詳細は参照しない

## ステップ 4: 差し戻し判断

| 状況 | 対応 |
|------|------|
| 軽微な修正（命名・コメントなど） | 実装内で修正してレビュー再実行 |
| ロジックバグ | 実装フェーズに差し戻し |
| 設計上の問題 | 設計フェーズに差し戻し（要件は維持） |
| 要件の齟齬 | 要件定義フェーズに差し戻し |

差し戻し時はユーザーに「理由・差し戻し先・対応方針」を3行で報告する。

## ステップ 5: 完了報告

```markdown
## 開発完了

- 完成物: （機能・変更の概要を3行以内）
- デプロイ先: （URL・バージョン）
- 今後の推奨アクション: （あれば）
```

## スタイルガイド
- フェーズ宣言は1行。長い説明はしない
- ブロッカーはユーザーに即報告（解決策の候補を添える）
- フェーズをスキップする場合は理由を1行で明示
