# Blog Creator
ユーザーと対話（壁打ち）しながらブログ記事を作るスキル。
完成物は HTML Artifact（ブラウザで見られるページ）として出力し、要望があれば Word にも変換する。

## ステップ 1: 方向性を固める（壁打ち）
以下の3点をユーザーに確認する。すでに明示されていればスキップしてよい。
1. **読者は誰か** — 開発者・ビジネス職・一般ユーザーなど
2. **記事のゴール** — インスパイア系／チュートリアル系／導入検討系 など
3. **カバー範囲** — 全体俯瞰／特定機能に絞る／体験談ベース など

確認したら、記事の骨格（タイトル案・セクション構成）を提示してユーザーの承認を得る。

## ステップ 2: 情報収集（必要な場合）
技術的なトピックや事実確認が必要な場合は `/deep-research` を呼び出して調査する。
調査結果の中から記事に使える知見を抽出し、正確な情報をベースにコンテンツを構成する。

## ステップ 3: デザイン方針を決める
`/artifact-design` のガイドラインに従い、以下を決定してから HTML を書き始める。
- **Color**: 4〜6色のパレット（背景・テキスト・アクセント・ボーダーなど）
- **Type**: 見出し用・本文用の2役以上のフォント。CDN不可のためシステムフォントを使う
- **Layout**: 1カラムのエディトリアルか、カード型か、ダッシュボード型かを選ぶ

デザインは subject（記事のテーマ・世界観）から導くこと。
AI 生成デザインの定番（クリーム＋セリフ＋テラコッタ、グラデーションヒーローなど）は避ける。

## ステップ 4: Artifact を作成する
HTML ファイルをスクラッチパッドに書き出し、`Artifact` ツールで公開する。
実装時の必須事項:
- 外部 CDN は使わない（CSP でブロックされる）
- `overflow-x: auto` で横スクロールを body に出さない
- `prefers-reduced-motion` を考慮する
- すべての要素を正しく閉じる

## ステップ 5: 壁打ちで磨く
Artifact を公開したらユーザーにフィードバックを求める。
内容・デザイン・トーンのいずれについても修正に対応し、都度 Artifact を更新する（同じファイルパスで再公開 = 同じ URL に上書き）。

## ステップ 6: Word 出力（オプション）
ユーザーが Word 出力を求めた場合、`python-docx` を使って `.docx` を生成し `SendUserFile` で渡す。
HTML のビジュアルを完全再現するのではなく、**読みやすい文書**として再構成すること。

## スタイルガイド
- 壁打ち中は短く、1〜2 文で問いかける
- ユーザーが「いいね」「OK」と言ったら即次のステップへ進む
- デザイン選択の理由を一言添えると納得感が出る
- 完成したら「次に直したい箇所はありますか？」で締める
